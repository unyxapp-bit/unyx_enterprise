# Unyx Enterprise / Unyx Ops
## README — Plano da Fase 3 (Validação com Cliente Real)

---

# 1. Objetivo da Fase 3

A Fase 3 tem como objetivo validar o Unyx Ops em um ambiente real de operação.

Depois da Fase 1, a base técnica estará pronta.

Depois da Fase 2, o MVP comercial estará demonstrável.

Na Fase 3, o foco será responder uma pergunta principal:

# O produto resolve uma dor real no dia a dia da operação?

Essa fase não é sobre adicionar muitas funcionalidades novas.

É sobre testar, observar, medir e ajustar.

---

# 2. Meta Principal

Ao final da Fase 3, o Unyx Ops deve estar validado com pelo menos um cenário real ou semi-real.

A meta é confirmar se o sistema realmente ajuda gerentes, fiscais, supervisores ou responsáveis pela operação a tomar decisões mais rápidas.

---

# 3. Público de Validação

A validação deve começar com negócios onde a dor operacional é clara.

## Segmentos prioritários

- supermercado
- loja de varejo
- farmácia
- restaurante

## Perfil ideal para testar

- gerente de loja
- fiscal de caixa
- supervisor de equipe
- líder operacional
- dono de pequeno negócio

---

# 4. O que será validado

## 4.1 Dor operacional

Validar se o cliente realmente sofre com:

- atrasos
- faltas
- intervalo mal controlado
- falta de cobertura
- troca de posto desorganizada
- sobrecarga da liderança
- dificuldade de saber quem deve sair primeiro
- falta de histórico confiável

---

## 4.2 Clareza do dashboard

Validar se o dashboard mostra rapidamente:

- quem está trabalhando
- quem está atrasado
- quem precisa sair
- quem está em intervalo
- quem voltou
- onde existe alerta crítico

---

## 4.3 Facilidade de uso

Validar se o sistema pode ser usado em uma rotina corrida.

Perguntas importantes:

- o fiscal entende a tela rapidamente?
- os botões estão claros?
- os status fazem sentido?
- a ordem de prioridade ajuda?
- o sistema reduz dúvida ou cria mais trabalho?

---

## 4.4 Valor comercial

Validar se o cliente pagaria pelo produto.

Perguntas importantes:

- isso resolveria um problema real?
- quanto isso economizaria de tempo?
- reduziria filas?
- reduziria erros?
- ajudaria a cobrar responsabilidade?
- teria uso diário?

---

# 5. Escopo da Fase 3

A Fase 3 será focada em validação.

## Entra na Fase 3

- teste com dados reais ou próximos da realidade
- simulação de operação diária
- feedback de usuários
- ajustes de tela
- ajustes de fluxo
- ajustes de status
- ajustes de prioridade
- melhoria de linguagem
- melhoria de usabilidade
- preparação de proposta comercial inicial

## Não entra na Fase 3

- gamificação
- feed corporativo
- comunidades
- IA avançada
- integração com ERP
- app mobile nativo
- cobrança automática
- relatórios complexos

---

# 6. Preparação para o Teste Real

Antes de validar com cliente, o sistema deve ter:

- login funcionando
- uma empresa cadastrada
- uma filial cadastrada
- colaboradores cadastrados
- escala do dia cadastrada
- dashboard carregando dados
- status operacional funcionando
- ações básicas registrando eventos
- histórico simples disponível

---

# 7. Dados para Validação

## Cenário mínimo

- 1 empresa
- 1 filial
- 2 ou 3 setores
- 10 a 20 colaboradores
- 1 escala diária completa
- alguns atrasos simulados
- alguns intervalos simulados
- 2 ou 3 alertas críticos

---

# 8. Fluxo de Teste Recomendado

## Teste 1 — Início do dia

Objetivo:

Validar se o sistema mostra a situação inicial da operação.

O usuário deve conseguir ver:

- quem deveria estar trabalhando
- quem ainda não confirmou entrada
- quem atrasou
- quem faltou

---

## Teste 2 — Controle de intervalo

Objetivo:

Validar se o sistema ajuda a decidir quem deve sair para intervalo.

O usuário deve conseguir:

- visualizar quem deve sair
- iniciar intervalo
- confirmar retorno
- ver status atualizado

---

## Teste 3 — Cobertura operacional

Objetivo:

Validar se o sistema ajuda a evitar posto descoberto.

O usuário deve conseguir identificar:

- quem está em intervalo
- quem está trabalhando
- onde existe risco de cobertura
- quais alertas são críticos

---

## Teste 4 — Histórico

Objetivo:

Validar se o histórico ajuda a liderança.

O usuário deve conseguir ver:

- eventos registrados
- horários reais
- quem executou a ação
- observações importantes

---

# 9. Roteiro de Apresentação para Cliente

## 1. Abrir com a dor

Exemplo:

"Hoje, quando alguém atrasa, falta ou precisa sair para intervalo, o gerente normalmente só descobre quando o problema já virou fila ou reclamação."

---

## 2. Mostrar o dashboard

Apresentar:

- visão geral
- atrasos
- intervalos
- alertas
- colaboradores em operação

---

## 3. Simular uma situação real

Exemplo:

- colaborador atrasado
- colaborador precisa sair para intervalo
- caixa aguardando cobertura
- retorno confirmado

---

## 4. Mostrar histórico

Mostrar que tudo fica registrado.

Isso reforça:

- controle
- rastreabilidade
- segurança gerencial

---

## 5. Perguntar sobre valor

Perguntas:

- isso ajudaria na sua rotina?
- qual parte mais chamou atenção?
- o que está faltando?
- você usaria isso todos os dias?
- quanto tempo isso economizaria?

---

# 10. Coleta de Feedback

Durante a validação, registrar:

## Sobre a dor

- qual problema mais incomoda?
- quando ele acontece?
- quem sofre mais com ele?
- como resolvem hoje?

## Sobre a tela

- entendeu rápido?
- quais cards fazem sentido?
- faltou alguma informação?
- tem informação demais?

## Sobre o fluxo

- quais ações deveriam ser mais rápidas?
- quais botões estão confusos?
- qual status não faz sentido?

## Sobre preço

- pagaria por isso?
- quem aprovaria a compra?
- seria mensal ou anual?
- valor por loja ou por empresa?

---

# 11. Métricas de Validação

A Fase 3 deve medir:

- tempo para entender o dashboard
- tempo para localizar um problema
- número de cliques para registrar uma ação
- clareza dos status
- interesse em testar novamente
- interesse em pagar
- sugestões recorrentes

---

# 12. Ajustes Esperados

Depois dos testes, provavelmente será necessário ajustar:

- nomes dos status
- cores dos alertas
- ordem de prioridade
- texto dos botões
- fluxo de intervalo
- fluxo de sangria
- tela de escala
- filtros por setor
- visual do dashboard

---

# 13. Critério de Sucesso da Fase 3

A Fase 3 será considerada bem-sucedida se:

1. O usuário entender o dashboard rapidamente.
2. O usuário reconhecer a dor como real.
3. O sistema ajudar a tomar uma decisão operacional.
4. O usuário sugerir melhorias práticas.
5. O usuário demonstrar interesse em usar novamente.
6. O produto parecer útil o suficiente para virar assinatura.

---

# 14. Resultado Esperado

Ao final da Fase 3, você deve ter:

- feedback real
- melhorias priorizadas
- fluxo operacional validado
- proposta de preço inicial
- argumentos comerciais mais fortes
- base para vender o primeiro piloto
